# Execution Plan: MVP

**Status:** In Progress
**Started:** 2026-03-12
**Target:** 2026-03-26

## Goal

Ship a useful first version of Orchestra that:
1. Registers multiple repos
2. Tracks work across them from one dashboard
3. Starts and observes Symphony-backed runs
4. Evaluates a repo for harness-readiness
5. Recommends the first hardening epic for brownfield repos

## Non-Goals

Not MVP:
- replacing Symphony's runner loop
- replacing Symphony's workspace model
- building a complete native tracker before the control plane is useful
- deep GitHub-native workflow as a prerequisite

## Milestones

### Week 1: Control Plane Skeleton

- [x] Project scaffolding
- [x] Database schema skeleton
- [x] Basic orchestrator skeleton
- [x] Initial docs and workflow shape
- [ ] Symphony adapter boundary
- [ ] Run model aligned with Symphony-backed execution
- [ ] Project registry UI

### Week 2: Visible Runs

- [ ] Start a Symphony-backed run from Orchestra
- [ ] Persist run status, timestamps, and logs
- [ ] Show active and historical runs in dashboard
- [ ] Persist key artifacts from each run
- [ ] Define workflow parsing/schema contract

### Week 3: Harness Readiness

- [ ] Repo scan for harness-readiness signals
- [ ] Evidence-backed findings report
- [ ] Recommended first hardening epic
- [ ] Create follow-up issue from audit output

### Week 4: Self-Hosted Story

- [ ] Docker Compose setup
- [ ] Single-container local deployment path
- [ ] README walkthrough
- [ ] Orchestra managing Orchestra work in a minimal loop

## Deliverables

### 1. Multi-project control plane

Must exist:
- add/list projects
- issue overview
- run overview
- project-specific workflow config

### 2. Symphony integration

Must exist:
- a clear adapter boundary
- one working execution path
- normalized run state in Orchestra
- durable logs/artifacts

### 3. Repo audit

Must exist:
- readiness score or equivalent graded output
- evidence for findings
- generated first epic

## Risks

### Risk: docs and code drift

Current code still contains native-runner assumptions.
Mitigation:
- keep docs explicit about temporary vs target state
- introduce adapter boundary before deeper implementation

### Risk: weak Symphony integration story

If Orchestra cannot expose useful run evidence, the dashboard is theater.
Mitigation:
- prioritize artifact ingestion early
- prove one end-to-end execution path

### Risk: audit becomes vague advice

"This repo is messy" is not a feature.
Mitigation:
- require findings to include evidence
- require output to produce a concrete first epic

## Immediate Next Steps

1. Define the Symphony adapter interface.
2. Parse `WORKFLOW.md` with an explicit schema.
3. Prove one issue -> run -> logs/artifacts path.
4. Design the first harness-readiness audit rubric.
